Java面试题

一、异常类结构图

  • Throwable：所有的异常都直接或间接继承Throwable，Throwable是所有异常的祖先
  • Error： 是程序中无法处理的错误，表示运行应用程序中出现了严重的错误。此类错误一般表示代码运行时JVM出现问题
  • Exception：可处理的异常，即程序本身可以处理的异常。
    • 非运行时异常：除RuntimeException及子类外的属于Exception类的异常，该编译器会检查此类异常，必须对该异常进行处理，要么使用try-catch捕获，要么使用throws语句抛出，否则编译不通过
    • 运行时异常：Runtime Exception及其子类表示JVM在运行期间可能出现的错误，编译器不会检查此类异常，并且不要求处理异常。一般时逻辑异常

二、JVM

1、JVM内存结构

  • 线程共有
    • 堆： 几乎所有的对象实例和数组都在堆分配内存。Java Heap 是垃圾收集器管理的主要区域
    • 方法区： 用于存储已经被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。
  • 线程私有
    • Java虚拟机栈： 是 Java 方法执行的内存模型：每个方法被执行的时候都会同时创建一个栈帧，而栈帧用于存储局部变量表、操作数栈、动态链接、方法返回地址和一些额外的附加信息
    • 本地方法栈：用于本地Native方法
    • 程序计数器：下一条执行的字节码指令

2、JVM堆+方法区

  • 堆
    • 新生代 1/3
      • Eden区 8/10：存放新生对象
      • Survivor区 2/10：存放Eden区垃圾回收后的存活的对象
        • from区 1/2：
        • to区 1/2：
      • 具体过程
        • Eden区+from区垃圾回收之后，把存活对象复制到to区，然后把to区，清空from区和Eden区，然后原from区改为to区，to区改为from区
    • 老年代 2/3： 一些生命周期较长的对象或者特别大的对象。如何判断生命周期较长：经过一定次数的垃圾回收后存活的对象，默认是15次
  • 方法区
    • 元空间：JDK1.8之后，元空间取代永久代，元空间并不在虚拟机中，而是直接使用本地内存 ，默认情况下，元空间大小仅受本地内存限制

3、对象是否被回收

  • 引用计数法：对象添加一个引用计数器，引用时+1，没被引用则-1，当引用计数器为0时，就会被垃圾回收
    • 优点：实现简单，执行效率高
    • 缺点：存在循环引用的问题。对象A，B相互引用，那么A，B的引用计数器永远不为0，那么就无法判断A，B是否要垃圾回收
  • 可达性分析算法：从GC ROOT向下搜索，而这过程是称为引用链，当对象跟GC ROOT没有引用链相连时，那么这个对象不可达，会被垃圾回收
    • GC ROOT：类加载器、Thread、虚拟机栈的本地变量表、静态成员变量
    • 被判为需要回收时，一定回收吗？
      • finalize()方法最终判定对象是否存活
        • 筛选：判断不可达对象finalize()方法是否被覆盖或者已被执行，如果是，就对象回收
        • 标记：筛选后判断是有必要执行finalize()方法，不过执行不是一定会执行完，因为对象存放在队列中，如果出现极端情况( 如死循环 )的话，就可能导致其他对象等待甚至内存回收系统崩溃
        • finalize()方法：实际上是让不可达的对象与可达对象关联，使其可达

4、垃圾收集算法

  • 标记-清除算法： 先标记出所有需要回收的对象，在标记完成后统一回收掉所有被标记的对象
  • 复制算法
  • 标记-整理算法
  • 分代收集算法
